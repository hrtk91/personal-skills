# review スキル

変更差分を、6つの観点でレビューして報告する。

目的は「サブエージェントを起動すること」ではなく、変更の欠陥・責務逸脱・追従漏れ・テスト不足を見つけること。小さな差分ではメインエージェントだけで簡易レビューしてよい。大きい差分、複数レイヤーにまたがる差分、認証/権限/課金/永続化/外部連携を含む差分では観点ごとに並列レビューしてよい。

## 手順

### 1. 変更ファイルの特定

- 指定がなければ `git diff --name-only main...HEAD` で変更ファイル一覧を取得する。
- base branchが `main` とは限らない。必要なら `git merge-base` やリモートdefault branchを確認する。
- 作業ツリーに未コミット差分がある場合は、`git diff --name-only` も確認する。
- 変更ファイルだけでなく、呼び出し元・型定義・ルーター・設定・テストも必要に応じて読む。

### 2. レビュー観点

#### 命名一貫性

確認すること:

- ドメイン用語、ユースケース名、API名、UI表示名が同じ概念に対して揃っているか。
- `my` / `authored` / `owned`、`submit` / `create` / `publish` のような近い言葉が混在していないか。
- Booleanや権限名が真偽両方を自然に表せるか。例: `with_admin(False)` のように読みにくくないか。
- ファイル名、関数名、型名、テスト名が責務や仕様変更を正しく表しているか。
- コメント、エラーメッセージ、テストspecが実装後の仕様とズレていないか。
- 既存の命名規則やディレクトリ配置に沿っているか。

#### レイヤー責務 / 依存方向

確認すること:

- domain / usecase / adapter / infrastructure / UI / test の責務が混ざっていないか。
- ドメインがadapterやUIの都合に依存していないか。
- ルーターやControllerにビジネス判断・認可ポリシー・永続化の詳細が漏れていないか。
- usecaseが複数repoや外部サービスをオーケストレーションする境界として機能しているか。
- repositoryやportの向きが逆転していないか。
- shared UIがAPI DTO全体に依存するなど、下位/共有レイヤーが上位の具体型を知りすぎていないか。
- Feature-Sliced Design等、プロジェクト固有のレイヤールールがある場合はそれに従っているか。
- 認証(Authentication)と認可(Authorization)の責務が混ざっていないか。
- 権限判定をJWTクレームだけに閉じるなど、現在状態を確認すべきルールを古い情報で判断していないか。

#### 型安全性

確認すること:

- 文字列・数値・Boolの生値で表すべきでない概念がVO、enum、union、Resultで表現されているか。
- DTOとdomain型の境界が明確か。
- APIレスポンスのschema/zod/decoderがbackend変更に追従しているか。
- null/undefined/Option/Resultの扱いが網羅的か。
- `as` や unchecked cast、panic、unwrap相当が増えていないか。
- バリデーションがUIだけでなくサーバ側/ドメイン側にもあるか。
- 権限や状態をBooleanで足してよいか。複数状態に増える可能性が高いならrole/permission型が必要ではないか。
- exhaustive matchが崩れていないか。

#### 保守性

確認すること:

- ネスト、重複、分岐、早期returnのバランスが読みやすいか。
- 既存パターンから外れた独自実装をしていないか。
- エラーハンドリング、ログ、HTTP status、ユーザー向けメッセージが既存方針に合っているか。
- 小さな変更に対して抽象化しすぎていないか。逆に共通化すべき重複を増やしていないか。
- テストヘルパーやfixtureが仕様を隠して読みにくくしていないか。
- 設定値、feature flag、環境変数のデフォルトが安全か。
- 将来の変更時に追従箇所が分かるか。コメントが有益か、古くなりやすいコメントを増やしていないか。

#### 影響範囲

確認すること:

- 変更した関数・型・DTO・route・環境変数・DBカラムの呼び出し元を `rg` 等で探索したか。
- backendを変えたらSPA/CLI/E2E/docs/schema/mock/seedsが追従しているか。
- API response/requestの変更がクライアント、テスト、ドキュメントに反映されているか。
- 認証/認可変更で、未認証・一般ユーザー・管理者・所有者・非所有者などの境界が全て意図通りか。
- UIのメニューだけでなく、直URL、CTA、空状態メッセージ、CLI案内、ブックマーク経由も追従しているか。
- 環境変数や設定追加がdev/e2e/prodで設定されているか。
- DB migrationや既存データへの影響があるか。
- 既存CI/E2Eが新仕様で落ちないか。

#### テスト充足性

確認すること:

- 正常系だけでなく、異常系・境界条件・権限差分・未認証・不正入力がテストされているか。
- 仕様変更の主目的がUI表示ならUIテスト、API制約なら統合/APIテスト、CLI表示ならCLI/E2Eテストがあるか。
- 「落ちてほしくない既存仕様」の回帰テストがあるか。例: admin必須化後も未認証は403ではなく401。
- backend変更に対してSPA/CLI/E2Eのテスト前提が古くなっていないか。
- テスト名や `@testspec` が新仕様を正しく説明しているか。
- fixtureがadminやownerなどの前提を暗黙にしすぎていないか。
- full suiteが重い場合でも、変更リスクに見合う最小コマンドを提示しているか。

### 3. 実行スタイル

- 小さな差分: メインエージェントが6観点をまとめて簡潔に見る。
- 中〜大きい差分: 観点ごとにサブエージェントを並列で起動してよい。
- 複数サブエージェントを使う場合でも、最終判断はメインエージェントが原典を確認して行う。
- 指摘は重大度順に並べる。

重大度:

- 重大: マージすると仕様破壊、セキュリティ問題、CI/E2E failure、データ破壊が起きうる。
- 中: 今すぐ壊れないが、責務逸脱、追従漏れ、テスト不足、将来のバグ温床になる。
- 軽微: 命名、コメント、読みやすさ、追加テストの改善提案。

### 4. 報告形式

指摘がある場合:

```text
## レビュー結果

- [重大] path:line — 内容
- [中] path:line — 内容
- [軽微] path:line — 内容

補足:
- 問題なしだった観点や、確認したが採用しなかった懸念を書く。
```

指摘がない場合:

```text
全観点で問題なし。

確認した観点:
- 命名一貫性
- レイヤー責務 / 依存方向
- 型安全性
- 保守性
- 影響範囲
- テスト充足性
```
