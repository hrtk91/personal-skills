# comment-processing-order

複数段階の処理を読む人が、コードを上から追う前に全体の流れを掴めるようにコメントを整える。

## 使う場面

- 関数が外部I/O、プロセス起動、DB/API呼び出し、ファイル操作、非同期処理、状態遷移、複数のエラー分岐を含む。
- ユーザーが「処理順を箇条書きして」「コードブロック前にどの処理か書いて」「コメント足りてない」と求めた。
- レビューで、コードの目的ではなく処理の順番や副作用が見えにくいと判断した。

## コメント形式

関数またはまとまった処理の冒頭に、処理順を番号付きで列挙する。

```text
// 処理順:
// 1. 入力を検証して正規化する。
// 2. 操作に必要な永続状態を読む。
// 3. 外部副作用を実行する。
// 4. 結果を保存して出力先を表示する。
```

各コードブロックの直前に、冒頭リストと同じ番号・文言に対応するコメントを置く。

```text
// 1. 入力を検証して正規化する。
```

日本語コードベース、またはユーザーが日本語で依頼している作業では、日本語コメントを優先する。

## 判断基準

- コメントは「何をするか」と「なぜその順番か」が分かる粒度にする。
- 1行ごとの説明や、コードを読めば自明な代入の説明は避ける。
- リスト項目は3から7個程度に収める。多すぎる場合は関数分割を検討する。
- 番号コメントと実コードの順番がずれたら、コメントではなく構造を先に疑う。
- コメント追加で古い説明が残る場合は、古いコメントを更新または削除する。

## 作業手順

1. 対象関数の副作用、入力、出力、エラー分岐を読んで処理段階を抽出する。
2. 冒頭に番号付きの処理順コメントを追加する。
3. 各段階を実行するコードブロック直前に対応番号のコメントを置く。
4. コメントと実際の処理が一致しているか、上から読み直して確認する。
5. フォーマットと既存テストまたはビルドを実行する。
