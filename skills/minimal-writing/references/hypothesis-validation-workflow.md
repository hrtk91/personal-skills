# 仮説検証とADR管理

この skill の文章ルールは、固定の好みではなく仮説として扱う。

## 分けるもの

- ルール: `SKILL.md` にある現在の実行方針。
- 仮説: そのルールが効くはずだという検証可能な主張。
- 評価ケース: 仮説を試すための入力・期待挙動。
- 評価結果: 実際に試した結果、失敗、差分、観察。
- ADR: 仮説、評価結果、代替案、副作用を踏まえて、採用・変更・棄却を決めた履歴。

ADR は検証結果そのものではない。ADR は、検証結果を踏まえた意思決定履歴である。

## 変更前に確認すること

skill のルールを変える前に、少なくとも次を明らかにする。

```text
仮説ID:
対象ルール:
問題になった文章ケース:
期待する改善:
失敗した場合の兆候:
評価ケース:
評価結果:
ADR:
```

## 受け入れ条件を書く

仮説には、採用条件と棄却・修正条件を両方書く。

例:

```text
H-XXXX: 各文の役割を明示してから圧縮すると、冗長な文が減る。

採用条件:
- 複数媒体で、削除しても読者の理解・判断・行動が変わらない文が減る。
- 根拠、制約、不明点は圧縮後も残る。

棄却・修正条件:
- 重要な制約や判断理由まで削られる。
- 文が短くなっても読者の判断が難しくなる。
```

## 評価結果に残すもの

評価結果には、結論だけでなく判断材料を残す。

- 入力文章または対象差分
- 適用したルール
- 出力またはレビュー結果
- どの指標が改善・悪化したか
- 採用しなかった修正案と理由

## ADRに残すもの

ADR には次を残す。

- 背景: どの失敗・不足から始まったか。
- 決定: どのルールを採用・変更・棄却するか。
- 根拠: どの仮説と評価結果にもとづくか。
- 代替案: 検討したが取らなかった選択肢。
- 結果: 良い影響と副作用。

## 最小運用

小さな改善では、専用の `hypotheses/` や `evals/results/` を必ず作らなくてもよい。ただし、ADR または eval case の中で、仮説、受け入れ条件、評価結果、決定理由が追える状態にする。
