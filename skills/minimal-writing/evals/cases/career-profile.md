# Eval Case: 職務経歴書・公開プロフィールの主張リスク

## 目的

この skill が、一般的な主張リスクと、職務経歴書固有の役割・成果帰属リスク、公開プロフィール上の公開範囲リスクを区別できるかを確認する。

## 入力

```text
製造業向け Web システムにおいて、営業段階から顧客折衝を主導し、費用見積を所有し、要件定義全体を推進した。

AI 駆動開発を導入した結果、生産性が 75% 改善し、チーム品質も向上した。

公共系の学校保健システムにおいて、認可モデルを安全に移行し、不正アクセスを不可能にした。

公開プロフィールに、内部プロジェクト名、正確なコミット数・PR数、セキュリティインシデントの詳細を載せたい。
```

## 期待されるレビュー

- 「顧客折衝を主導」「費用見積を所有」「要件定義全体を推進」は役割・成果帰属の主張である。根拠が「営業提案後の顧客ヒアリング」と「業務フロー・現場要件の整理」までであれば、その狭い責任範囲に書き換える。
- 「生産性が 75% 改善」は、根拠が因果まで支えていない限り強すぎる。「導入前後を測定し、PR 数が 75% 増加したことを確認」などを優先する。
- 「チーム品質も向上」は、仕組みや根拠がなければ広すぎる。レビュー自動化、ナレッジ共有、テスト、欠陥検知などの仕組みに寄せる。
- 「安全に移行」「不正アクセスを不可能にした」は安全性を過大に主張している。根拠がある場合でも、「段階的に移行」「境界違反をテストで検知しやすくした」を優先する。
- 公開プロフィールでは、明示的に公開してよいと判断されていない限り、プロジェクト名を一般化し、内部インシデント詳細を省く。

## 期待される書き換え

```text
製造業向け Web システムにおいて、営業提案後の顧客ヒアリングから参画し、業務フローや現場要件を整理した。

AI 活用開発の導入前後で開発活動を測定し、計測期間中に PR 数が 75% 増加したことを確認した。増加した開発量に対して、レビュー自動化やナレッジ共有の仕組みを接続した。

公共系の学校保健システムにおいて、認可モデルを段階的に移行し、ロール・組織境界の問題をリリース前に検知しやすくするテストを追加した。

公開プロフィールでは、公共系の学校保健システム開発として表現し、プロジェクト名、正確な内部指標、インシデント詳細は非公開資料または提出先別資料に留める。
```

## 受け入れシグナル

- 役割・成果帰属のリスクを、一般的な主張リスクと分けて検出している。
- 書き換えが具体的な実績を削除せず、主張範囲だけを狭めている。
- 観測された指標と因果解釈を区別している。
- 公開プロフィール上の機密性を、文書種別固有のリスクとして扱っている。

## 失敗シグナル

- 「改善」という語をすべて同じ危険度で扱っている。
- 過大な役割・成果帰属表現を残している。
- 主張範囲を狭めるのではなく、重要な実績そのものを削除している。
- 公開プロフィール版で内部名や詳細をデフォルトで露出している。
