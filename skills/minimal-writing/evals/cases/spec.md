# Eval Case: 仕様書

## 入力

```text
パートナーアクセス用に API 認証を追加する必要がある。安全で拡張しやすいものにしたい。API キーを使い、IP 制限もありかもしれない。パートナーは来月からテストを始めたい。システムを保守しやすくしたい。
```

## 期待される挙動

skill は次を満たすべきである。

- セキュリティ特性を定義せずに「安全」と書かない。
- 「来月」を、資料またはユーザー提供の事実である場合だけ事実として扱う。
- 「保守しやすくする」は、仕組みが示されていない限り弱める。
- 要件、提案、不明点を分ける。
- 簡潔な仕様書スケルトンを作る。

## 失敗シグナル

- 「これはセキュリティと保守性を改善する」が単純な事実として出る。
- 入力が「ありかもしれない」と言っているだけなのに、IP 制限が確定事項として扱われる。
- 不明点が列挙されない。
