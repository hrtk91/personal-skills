# Minimal Writing

文章を「読者の状態を目的の状態へ移すための、必要十分で最小の情報構造」として扱う skill。文章をきれいにするだけではなく、情報の過不足、文の役割、主張の根拠、不明点、判断理由を検品する。

## 解く問題

次の問題がある文章作業で使う。

- 不要な文、重複、一般論、雰囲気だけの前置きを削りたい。
- 根拠のない主張や過大表現を避けたい。
- 事実、推論、意見、提案、不明点を分けたい。
- 因果、安全性、改善、保守性、将来効用の言い切りを点検したい。
- 長い文章を、根拠・制約・不明点を消さずに短くしたい。
- 媒体ごとの期待値に合わせて、文章の構造・削る情報・残す情報を切り替えたい。

これは汎用の文体ガイドではない。単なる言い換え、翻訳、敬語調整だけで主張の検証が不要な場合は使わない。コードレビューや設計レビューのように、文章品質ではなく実装リスクが主対象の作業では、対象分野の skill を優先する。

職務経歴書で「状況 / 判断と行動 / 結果」のエピソード設計が主目的の場合は `career-accountability-review` を優先し、主張の強さや根拠の扱いを点検したい場合にこの skill を併用する。

## 中核の考え方

> 文章とは、特定の読み手を特定の理解・判断・行動へ動かすための、必要十分で最小の情報構造である。

したがって:

1. 役割のない文はノイズである。
2. 根拠のない主張を事実として書いてはいけない。
3. 不明点は、もっともらしい推測で埋めず、見える状態にする。
4. 読み手に見せる本文と、判断の根拠・検討過程は分ける。
5. 文章ルールは仮説であり、レビュー、指標、ADRによって改善する。

## 主張の種類

重要な主張は、執筆前またはレビュー時に分類する。

- 事実: 提供された資料、ユーザーが示した事実、リポジトリ内容、データ、引用可能な参照で確認できる事実。
- 推論: 事実から導いた推論。ただし直接証明されたものではない。根拠と不確実性を示す。
- 意見: 評価、好み、解釈、判断。
- 提案: 推奨する行動、設計、方針。
- 不明: 未確認の情報。事実であるかのように本文へ入れない。

根拠が提供されていない場合、根拠を捏造しない。主張を不明として扱う、必要な場合だけ質問する、または明示的な仮定・プレースホルダー付きで下書きする。

## 文の役割

すべての文は、明確な役割を1つ持つべきである。代表的な役割は次のとおり。

- 結論
- 読み手への前提提示
- 要件
- 制約
- 根拠
- 例
- 手順
- 警告
- 例外
- トレードオフ
- 判断理由
- 未解決の問い
- 次の行動

削除しても読み手の理解、判断、行動が変わらない文は、削除または統合する。

## デフォルトワークフロー

ユーザーが最終文だけを求めている場合を除き、次のフェーズで進める。

### 1. 執筆ブリーフを作る

執筆前に、必要最小限のブリーフを推定または確認する。十分な文脈がある場合は質問で止めず、仮定を明示して進める。

必要に応じて、内部または出力で次の形式を使う。

```text
文書種別:
読み手:
読み手の目的:
書き手の目的:
読後に読み手が到達すべき状態:
使用してよい事実:
使用してよい推論:
禁止する主張・トーン:
必要なセクション:
長さの制約:
トーン:
不明点:
```

### 2. 主張台帳を作る、または監査する

重要な主張について、台帳を作る。

```text
| ID | 主張 | 種類 | 根拠 | 読み手に必要か | リスク | 表現ルール |
|---|---|---|---|---|---|---|
```

ルール:

- 事実には根拠または提供された情報が必要。
- 推論には根拠と不確実性の表示が必要。
- 意見を中立的な事実のように見せない。
- 提案には理由またはトレードオフを添える。
- 不明点は「不明点 / 未解決の問い」に置き、本文で事実扱いしない。

### 3. トピックではなく機能でアウトラインを作る

意味を持つ機能的な見出しを優先する。

悪い例:

```text
認証
```

良い例:

```text
パートナー向けリクエストには API キー認証を使う
```

悪い例:

```text
エラー処理
```

良い例:

```text
再試行できるエラーと即時失敗を分ける
```

### 4. 不確実性を制御して下書きする

執筆時は次を守る。

- ジャンル上の理由がない限り、結論を早めに置く。
- 読み手の行動を変えない一般論は追加しない。
- 「保守性が向上する」のような将来効用の主張は、仕組みを示せない限り使わない。
- 重要な制約、エッジケース、不明点、判断理由を残す。
- 媒体に合うトーンにする。ただし、事実・推論・意見・提案・不明点の区別を、見栄えのために崩さない。

### 5. レビューを実行する

次の順序でレビューする。

1. **必要性レビュー**: 各文は理解、判断、行動に影響するか。影響しないなら削除または統合する。
2. **事実性レビュー**: 事実、推論、意見、提案、不明点が分離されているか。
3. **構造レビュー**: 読み手の判断経路に沿った順序になっているか。
4. **圧縮レビュー**: 根拠、制約、不明点、判断理由を消さずに短くする。
5. **リスクレビュー**: 誤解、過大表現、根拠より強い確信を与える箇所を特定する。

### 6. 適切な形で出力する

単純な書き換え・下書き依頼では、最終文と必要最小限の仮定メモだけを出す。

高リスク文書、媒体別判断が必要な文書、または検証過程を残したい依頼では、必要に応じて次を含める。

- 最終文
- 不明点 / 未解決の問い
- 主張台帳
- レビューNote
- ADR

プロセス成果物は、有用な場合だけ出す。不要に増やさない。

## 媒体別の調整

媒体別の期待値は `references/document-type-techniques.md` を読む。`SKILL.md` 本体には、媒体に依存しない普遍原則だけを残す。

## skill 自体の PDCA

この skill を仮説として扱う。

### Plan

検証する文章ルールの仮説を書く。

例:

```text
H-001: 下書き前に主張台帳を必須にすると、LLM が生成する文章の根拠なし主張を減らせる。
```

### Do

保存した入力・出力ペアを使って、複数の文書種別に skill を適用する。

### Check

次の指標で評価する。

- 根拠なし主張率
- 冗長率
- 曖昧さの数
- 不明点の扱い
- 目的適合
- 構造適合
- 圧縮安全性

### Act

ルール変更の理由を説明する ADR とともにのみ、skill を更新する。

まだ「提案中」のルールは、暫定的なレビュー観点として扱い、その旨を明示する。ADR の受け入れ条件を満たすまでは、採用済みの文体ルールとして扱わない。

仮説、評価ケース、評価結果、ADR、skill ルールの関係を追跡するときは `references/hypothesis-validation-workflow.md` を読む。

## この skill の改善を依頼されたとき

1. 失敗した文章ケースを特定する。
2. 失敗を分類する: 根拠なし主張、冗長性、文脈不足、構造不備、トーン不一致、隠れた不明点、過圧縮、ジャンル不一致。
3. ルール変更案を出す。
4. ADR を追加または更新する。
5. eval case を追加または更新する。
6. `SKILL.md` は簡潔に保ち、詳細は `references/` に移す。

テンプレートと詳細チェックリストは `references/` を参照する。軽量なヒューリスティックレビューが有用な場合は `scripts/review_text.py` を使う。
