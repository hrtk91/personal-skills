# career-accountability-review スキル

職務経歴書や転職プロフィールを、「何をしたか」だけでなく「どの状況で、何をどう判断し、何が変わったか」が伝わる形に整える。

## 最優先の型

成功・改善エピソードはこの3点で書く。

```text
状況: 何が問題で、なぜ放置できなかったか
判断と行動: 何を根拠に、何を選び、どう動いたか
結果: 何が改善し、チーム・プロジェクト・顧客・運用にどう効いたか
```

失敗・制約からの学びはこの3点で書く。

```text
状況: どんな制約・期待・納期・技術的不確実性があったか
行動と結果: 何を行い、どんな結果や問題が起きたか
失敗と学び: 原因をどう捉え、現在の判断基準にどう反映されたか
```

## 使うエピソードの種類

### 技術的な問題への対応

向いている材料:

- API、ドメインモデル、認可、帳票、テスト、CI、依存境界などが開発効率や品質を下げていた
- 将来の利用頻度、ロードマップ、リリースリスクから早期対応を判断した
- 分割、標準化、自動テスト、機械チェック、設計見直しにより保守性や分析容易性が上がった

書き方:

```text
状況: <対象> が <問題> を起こし、<影響> につながっていた。
判断と行動: <今後の利用頻度/リスク/期限> を踏まえ、<選択した方針> と判断。<具体的な実装・提案・合意形成> を行った。
結果: <保守性/品質/速度/障害分析/委譲> が改善し、<後続リリースやチーム運用> に効いた。
```

### 問題提起と合意形成

向いている材料:

- 性能、品質、セキュリティ、運用、レビュー負荷などの問題が継続していた
- 根本原因や実測データを整理し、上長・PM・専門チーム・他部署に提案した
- 予算獲得に限らず、方針合意、優先順位変更、別チーム連携、マージブロック導入も含めてよい

書き方:

```text
状況: <問題> が継続し、<現場/品質/納期/顧客> に影響していた。
判断と行動: <原因分析/実測/リスク整理> を行い、<関係者> に <改善案> を提案・調整した。
結果: <提案/方針> が採用され、<改善内容> により <定量/定性の効果> が出た。
```

### 失敗からの学びと成長

向いている材料:

- 技術選定、見積もり、チーム体制、品質担保、納期、レビュー運用などで想定外の困難があった
- 結果が完全成功ではないが、以後の判断軸が明確になった
- 他責ではなく、自分が次にどう判断するようになったかを説明できる

書き方:

```text
状況: <プロジェクトの性質> により、<期待/納期/品質> が求められていた。
行動と結果: <当時の判断や対応> を行ったが、<問題> が発生し、<影響> が出た。
失敗と学び: 原因は <見通し/体制/技術選定/品質設計> にあったと捉え、以後は <現在の判断基準> を重視している。
```

## レビュー観点

- 責任範囲: 自分が引き受けた範囲が明確か。参加、支援、担当、主導が混ざっていないか。
- 判断: 当時の制約、リスク、優先順位、選ばなかった案が見えるか。
- 行動: 何を作り、どこへ組み込み、誰と合意したかが具体的か。
- 結果: 数値、リリース、評価、委譲、運用改善、再発防止などの影響があるか。
- 学び: 失敗や制約を他責にせず、現在の判断基準に接続できているか。

## 表現の調整

- `主導` が強すぎるなら `担当`、`推進`、`支援`、`整理`、`提案` に落とす。
- `成果` が本人単独に見えるなら `土台を整備`、`一因となった`、`支援した` にする。
- 数値の定義が曖昧なら、測定対象を明記するか、外向き資料では数値を出さない。
- 職務経歴書本文でも、読み手に強く伝えたいエピソードは `状況` / `判断と行動` / `結果` のラベルを明示する。短い補足実績だけ自然文や通常の箇条書きにする。

## 報告形式

```text
## レビュー結果

- [高] path:line -- 状況、判断と行動、結果のどれかが欠け、過大表現や説明不足になっている
- [中] path:line -- エピソード化できる材料があるが、現状は抽象的
- [低] path:line -- 表現調整、順序、補足でさらに伝わりやすくなる

書き換え案:
- 状況: ...
  判断と行動: ...
  結果: ...
```

問題がない場合も、残るリスクや面接で補足すべき点を短く書く。
