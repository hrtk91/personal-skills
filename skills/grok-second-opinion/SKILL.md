# Grok Second Opinion

Grok 4.5を、結論を決める人ではなく別の角度から確認する相手として使う。

## 実行環境による使い分け

- 現在の実行主体がGrokなら、ユーザーがCLIでの別セッションを明示しない限り、現在のセッションで検討する。
- CodexやClaudeなどGrok以外から使う場合は、このスキルの手順でGrok CLIを起動する。

## 使う場面

- ユーザーがGrokへの相談やレビューを明示した
- 自分で一度調査したが、設計の選択に迷いが残る
- 原因の候補が複数あり、同じ見方を繰り返している
- レビューの見落としや思い込みを減らしたい

単純な置換や、まだ何も調べていない問題では自律的に起動しない。まず対象を確認する。

## 手順

1. 対象の実物を確認する。リポジトリなら現在のブランチ、未コミット変更、比較元、対象ファイルを調べる。
2. [context-guide.md](references/context-guide.md)に沿って、Grokへ渡す文脈を`/tmp`のMarkdownファイルにまとめる。試したことと結果、検討したが選ばなかった案と理由も含める。
3. リポジトリのルートを`--cwd`にして、OSの読み取り専用sandboxと読み取り用ツールだけで実行する。PRレビューでは`--diff-base`を渡し、ラッパーに差分の読み取り用コピーを作らせる。

```bash
<このスキルのディレクトリ>/scripts/run.sh \
  --cwd /path/to/repository \
  --prompt-file /tmp/grok-second-opinion.md \
  --diff-base origin/main
```

未コミット変更も対象なら`--include-uncommitted`を追加する。含めるかどうかはプロンプトにも明記する。

4. Grokの指摘を原典、コード、テスト結果と照らす。
5. 結果を「採用」「不採用」「追加確認が必要」に分け、自分の判断としてユーザーへ返す。

## 実行ルール

- 既定モデルは`grok-4.5`。別モデルへ黙って切り替えない。
- 既定は`--sandbox read-only`、ファイルの読み取り・検索・一覧だけ、`--no-memory`、`--no-subagents`、Web検索なし。編集とshellのツールはGrokへ渡さない。
- headless実行で読み取り確認待ちにならないよう`--always-approve`を使う。書き込み防止は許可確認ではなくsandboxとツール制限で担保する。
- `--permission-mode plan`は使わない。計画だけで終了し、差分レビューまで進まない場合がある。
- 最新情報や外部仕様の確認が必要な時だけ`--web`を付ける。その場合も一次資料を優先するよう依頼する。
- 認証情報、個人情報、未公開情報、対象外のファイル内容は渡さない。
- 長い会話全文や巨大なログを貼らない。Grokが作業場所から読めるファイルや差分は、場所と読み方を伝える。
- 試行錯誤は関係するものだけを「何をしたか」「どうなったか」で短く書く。失敗ログをそのまま並べない。
- 選ばなかった案は理由も書く。ただし、その理由が弱ければ指摘し、必要なら案を再検討するようGrokへ求める。
- 望む結論へ誘導しない。自分の案は「暫定案」と明記し、反例や別案を求める。
- Grokの回答をそのまま採用しない。指摘箇所と根拠を自分で確認する。

## 失敗時

- `grok-4.5`が使えない場合は`grok models`と認証状態を確認し、勝手に別モデルへ落とさずユーザーへ伝える。
- セッション保存先の権限エラーが出た場合は、Grok CLIがホーム配下へ書き込める実行権限で再実行する。
- 対象範囲が決められない場合は、推測で全体を渡さず、先に比較元や対象ファイルを確定する。
