# investigating-domain

## このスキルが補完すること

「調査して」「まとめて」という指示だけでは、subagent は:
- それっぽいファイルだけ選んで斜め読みする
- 事実と推測を混ぜた結論を出す
- カバレッジが保証されない

このスキルはその不足を補完する。**3フェーズの構造**と**subagent への指示フォーマット**がセット。

## 前提: ユーザー視点は常に必要

**システムのユースケースはユーザーのアクションから始まる。**
フロントエンドが「このボタン → このAPI → 画面遷移 → 次の操作」という流れを持つ結節点であり、バックエンドは渡された役割をこなすだけでユースケースの文脈を持っていない。

ユーザー視点を抜いた調査は、影響範囲からフロントエンドが漏れる原因になる。**両モード共通で、調査の最初にユーザー操作の起点を押さえること。**

> 誰が・どの画面で・何をするとこれが動くか

## 調査モード

モードの違いは「ユーザー視点を入れるかどうか」ではなく、**調査の広さ**だけ。

### モードA: ドメイン単体調査
ユーザー操作の起点を押さえた上で、特定モデル・ユースケースの内部構造や遷移を深掘りする。

### モードB: ビジネスフロー影響範囲調査
ユーザー操作の起点を押さえた上で、フロー全体を横断してトレースし仕様変更の影響範囲を割り出す。

## ワークフロー

### Phase 1: 概要調査（自分でやる）

**目的**: 詳細調査すべきファイルリストを確定する。結論は出さない。

**モードAの場合**:
1. `pages/` から調査対象を呼び出すユーザー操作の起点（画面・ボタン・API呼び出し）を特定する
2. Grep/Glob で変更対象の直接関連ファイルを特定する
3. 呼び出し元・呼び出し先を1階層だけ確認する

**モードBの場合**:
1. `pages/` 配下のロール別ディレクトリ（例: `pages/admin/`, `pages/user/`）から調査起点のページを特定する
2. そのページの API 呼び出し（`$api.xxx()` 等）と画面遷移先を1階層だけ追う
3. 「画面 → API → 画面」の地図をステップとして列挙する（これが調査対象ファイルリストになる）
4. 各ステップに対応するバックエンドの usecase / model を Grep で特定する

**出力**: ファイルパス一覧（ステップ対応表も可）。推測・結論は書かない。

**Phase 1 の追加ステップ: ドメイン仮説を立てる**

ファイルリスト確定後に、このフローで「画面表示・取得フィルタリング・状態遷移」を制御しているドメインモデルの仮説を列挙する。外れてもよい。Phase 2・3 で確認する。

例：
> - `SharingPermission` — 閲覧可否・ロール別表示を制御していそう
> - `Notification` — 通知ステータスで表示切り替えが起きていそう

この仮説を Phase 2 の subagent 指示に含めることで、ドメイン観点の報告漏れを防ぐ。

---

### Phase 2: 詳細調査（subagent に委譲）

**目的**: Phase 1 のファイルを1ファイルずつ Read して事実を収集する。

**鉄則**:
- ファイルを斜め読みしない。Read で全体を読む（これが「まとめて」との最大の違い）
- subagent には「事実のみ報告。推測・結論は書かない」と明示する
- 結論はメインスレッドで出す

**規模による subagent 分割**:

| ファイル数 | 分割方針 |
|-----------|---------|
| 5以下 | 1 subagent |
| 6以上（モードA） | データフロー軸（書き込み側 / 読み取り側）で分割 |
| 6以上（モードB） | ビジネスフローのステップ単位で分割 |

**subagent への指示テンプレート**:
```
以下のファイルを Read ツールで1ファイルずつ全体を読んでください。
推測や結論は不要です。各ファイルについて以下を報告してください:
- ファイルパスと主な責務
- 関連するモデル・クラス・関数の定義と引数
- 呼び出し元・呼び出し先(コード中に見えるもののみ)
- 気になった点(事実ベースで)

**ドメインモデルの制御観点（必須）**:
以下のモデルがファイル内で使われていれば、「何のクエリパラメータ・ステータス値・条件で画面表示やデータ取得を制御しているか」を事実ベースで報告してください:
<Phase 1 で立てた仮説モデルをここに列挙>

例: SharingPermission が使われていれば「status:'pending' でフィルタしている」「(resource_id, org_id) で存在チェックしている」等

[モードBの場合は追加]
このファイルが担当するビジネスフローのステップと、
前後ステップとの接続点（入力・出力となるモデル/関数）も報告してください。

対象ファイル:
- <path1>
- <path2>
```

---

### Phase 3: 統合（メインスレッドで実施）

1. subagent の事実報告を突き合わせる
2. 矛盾・不明点があれば**自分で Read して確認**する（subagent に再依頼しない）
3. 事実を構造化して出力する
   - モードA: 遷移フロー・副作用・冪等性などをステップ順に整理
   - モードB: ビジネスフローのステップごとに「担当ファイル・入出力モデル・処理内容」を並べる
   - **影響範囲の判断はしない**。事実を読んだ人が「ここを変えるとここに波及する」とわかる粒度で出す

4. **ドメインモデル × 画面制御の横断整理（必須）**

   Phase 1 の仮説モデルについて、以下の観点で整理する:

   | モデル | ユーザー側での制御 | 管理者/他ロール側での制御 | ステータス/条件の変化タイミング |
   |--------|--------------|----------------------|-------------------------------|
   | XxxModel | ... | ... | ... |

   これを怠ると「ファイルは読んだが重要な制御ロジックが漏れる」原因になる。

---

## よくあるミス

- Phase 1 で結論を出す → ファイルリストだけ
- **Phase 1 でドメイン仮説を立てない → subagent 指示にドメイン観点が入らず、画面制御の重要ロジックが報告されない**
- subagent に結論を出させる → 事実報告に徹させる
- Grep 結果だけで判断する → 必ず Read で全体を読む
- 矛盾を subagent に再調査させる → 自分でコードを確認する
- **Phase 3 でドメインモデル × 画面制御の横断整理をスキップする → 「ファイルは読んだが勘所が伝わらない」ナレッジになる**
