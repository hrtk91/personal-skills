# Zettelkasten (TIL)

体験した学びを永続的なナレッジとして記録する。デイリーノートに埋もれさせず、独立したノートとして切り出す。

## 保存先

- TILノート: `~/obsidian-vault/TIL/<タイトル>.md`
- デイリーノート: `~/.claude/skills/obsidian/scripts/daily-path.sh` で取得

## ワークフロー

1. ユーザーの学びを聞く(何が起きた？何を学んだ？)
2. タイトルを**主張/結論形式**で決める(検索しやすくするため)
3. TILノートを作成
4. デイリーノートの「学んだこと / メモ」セクションにリンクを追記
5. 既存TILとの関連があれば `## 関連` にリンクを追加

## タイトルの付け方

**主張/結論形式** にする。「何が」ではなく「どうだった」まで含める。

- 「Terraformはコードとstateがズレた状態でapplyすると破壊的なエラーが出る」
- 「Vueのcomputed内で副作用を起こすと無限ループする」
- 「PostgreSQLのpartial indexは WHERE句にマッチする行だけ索引するのでunique制約と組み合わせると強い」

## ノートのフォーマット

```markdown
---
tags:
  - <技術タグ>
  - til
created: YYYY-MM-DD
---

# <タイトル(主張/結論形式)>

## 問題

何が起きたか、何が問題だったか

## 原因

なぜそうなるのか

## 解決策 / 教訓

どうすればよいか

## 関連

- [[関連ノート]]
```

## デイリーノートへのリンク追記

「学んだこと / メモ」セクションの末尾に追記する:

```markdown
- [[タイトル]] #タグ1 #タグ2
```

## 関連ノートの探し方

既存TILを確認して関連があればリンクする:

```bash
ls ~/obsidian-vault/TIL/
```

## Claudeからの能動的なサジェスト

ユーザーが「TIL」と言わなくても、以下のような瞬間を察知したらClaudeから提案すること:

- バグの根本原因が判明したとき
- 意外な仕様・ライブラリの振る舞いを発見したとき
- ハマった問題を解決したとき（特に再現しにくいもの）
- 「知らなかった」「これは覚えておくべき」と感じた知見が出てきたとき

**サジェスト方法: `AskUserQuestion` を使う**

```
AskUserQuestion:
  question: "今回の「<学びの概要>」、TILとして残しておきませんか？"
  options:
    - label: "残す" → そのままzettelkastenワークフローへ
    - label: "スキップ" → 記録しない
```

テキストで聞くより選択肢UIの方が邪魔にならず、ユーザーが一瞬で判断できる。

## 注意事項

- 体験直後に書くのがベスト(時間が経つと曖昧になる)
- タグは具体的に(#terraform, #vue, #django など。#技術 のような抽象タグは避ける)
- 1ノート1テーマ。複数の学びがあれば分割する
