# Evaluation Coach 詳細ワークフロー

SKILL.md が500行を超えそうなときに参照する詳細ガイド。

## evidence 命名規則

```
evidence/<Objective-dir>/YYYY-MM-DD_<短い名前>.md
```

- 短い名前は kebab-case 推奨
- 同日複数なら `_01` `_02` を末尾に付ける
- ファイル名にKR番号を含めない（front-matterで管理）

## front-matter 必須項目

```yaml
---
date: YYYY-MM-DD          # 実際の活動日（PRマージ日や調査日。記録日ではない）
objective: O1             # O1/O2/O3/O4
key_result: KR1.1         # 複数なら配列も可: [KR1.1, KR3]
source: pr|commit|doc|meeting|notion|slack|daily|prm
tags: ["#KR1.1"]          # obsidianタグとして使える
---
```

## 良いevidenceの条件

1. **何を決めたか** が明示されている
2. **なぜそう決めたか** の判断基準が残っている
3. **やらないと決めたこと** も書かれている（スコープ管理）
4. **次に何を確認すれば前に進むか** が分かる
5. **他者が確認できる成果物** にリンクされている（PR / Notion / docs）
6. **再利用観点** が書かれている（他案件への流用可能性）

## 弱いevidenceのパターンと補強方法

| 弱い表現 | 補強 |
|---|---|
| 「〜を調査した」 | 何を対象にしたか / 何を対象外にしたか / 設計判断 / 次フェーズへの繋ぎ |
| 「〜を実装した」 | 採用した設計 / 比較した選択肢 / トレードオフ / レビューしてもらった人 |
| 「〜を支援した」 | 誰に対して / どう判断を委ねたか / 結果どう動いたか |
| 「〜を発表した」 | 誰宛か / 反応 / 次のアクションに繋がったか |

## 時期別フォーカス

### 期前半（〜2ヶ月）
- 運用リズム作り（週次レビューを定着させる）
- evidence型の整備（テンプレを実情に合わせて更新）
- 最初の意思決定ログを残す
- 主要案件の記録を始める

### 期中盤（2〜4ヶ月）
- 判断を明確にする
- 中間成果を出す
- デモ・手順・チェックリストに変換
- 他部署/他案件に渡せる形にする
- 継続/pivot/hold の判断材料を残す

### 期後半（4〜6ヶ月）
- KRカバレッジ不足を埋める
- 評価向けにサマリ化する
- 最終判断を明示
- 効いた学び/効かなかった学びを整理
- 評価文に転用できる短い証拠を集約

## 残業時間集計の連動

`obsidian` スキルの `scripts/work-hours.sh` を活用:

```bash
# 月単位の労働時間集計
bash ~/.claude/skills/obsidian/scripts/work-hours.sh --month 2026-04
```

集計結果を `評価.md` の残業時間表に転記する。
月45時間を超えた月があれば、原因（特定案件の集中 / 他要因）を備考に残す。

## 目標更新の判断基準

軽々しく目標を変えない。以下をすべて満たすときに変更候補を出す:

1. 現行KRが評価不可能になっている（測定方法が消失、対象事業が変わった等）
2. 実際の主戦場と明確にずれている
3. 重要な成果がKRから漏れている
4. 活動量のカウントに寄りすぎている

変更案を出すときは:
- 現行目標の問題
- 変更が妥当な理由
- 変更案
- 評価しやすさがどう改善するか
- 副作用や注意点

をセットで提示する。
