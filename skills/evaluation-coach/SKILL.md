# Evaluation Coach スキル

## 目的

半期評価は期末に書くものではなく、**期中に育てるもの**である。
朝のゴール設定から夜の振り返り、週次レビュー、evidence蓄積まで、**1日〜半期の成長サイクル**を一貫して伴走する。

ユーザーは「振り返りやろう」「今日のノート作って」「今期どう？」と言えばよい。
このスキルが自律的にデータ収集・KR接続・評価草案更新まで進める。

## 基本方針

- **収集は自動、解釈は対話**: git log・PR・ActivityWatchは機械的に集めるが、KR接続・判断の言語化はユーザーとの対話で行う
- **evidenceにないことは書かない**: 盛らない。証拠がなければ「今は弱い、次に何を残すか」を提案する
- **時期に応じた助言**: 期の前半・中盤・後半で必要な行動は違う。現在日付を見て判断する
- **過小評価を補正**: ユーザーは過小評価しがち。事実ベースで「これは強い実績」と伝える

## 設定

- Vaultパス: `~/obsidian-vault`
- デイリーノート: `~/obsidian-vault/デイリーノート/YYYY/MM/YYYY-MM-DD.md`
- 評価面談対策: `~/obsidian-vault/評価面談対策/<期>/`

### パス取得

```bash
~/.claude/skills/obsidian/scripts/daily-path.sh           # 今日のデイリーノートパス
~/.claude/skills/obsidian/scripts/daily-path.sh yesterday  # 昨日のパス
```

## ファイル構成

```
~/obsidian-vault/評価面談対策/<期>/
├── .coach.yaml           # 期/目標/締切/Objective設定
├── 目標設定.md           # 目標・KR本体
├── 評価軸メモ.md         # 評価軸
├── 評価.md               # 期末評価草案（期中に育てる）
├── data/
│   └── work-hours.jsonl  # 労働時間（日次蓄積）
├── docs/                 # 自己分析・効果測定ドキュメント
└── evidence/
    ├── _template.md
    ├── O1-技術基盤/
    ├── O2-PoC/
    ├── O3-リーダーシップ/
    └── O4-自己革新業務改善/
```

期は `scripts/current-period.sh` で自動判定（4-9月=H1、10-3月=H2）。

## モード一覧

ユーザー発話 → 内部でどのモードを実行するかを判断する。

### Mode M: 朝のルーティン (morning)

**いつ**: 「今日のノート作って」「デイリー作成」「今日のタスク確認して」、毎朝セッション開始時

**やること**:

1. デイリーノートパスを取得し、ファイルが無ければテンプレートから作成（親ディレクトリも `mkdir -p`）
2. 直近3営業日のデイリーノートを Read し、`- [ ]` 未完了タスクを収集
3. 昨日のデイリーノートの「作業ログ」と「優先タスク（[x]完了分）」から**朝会用サマリ**を構成
   - 親項目は `- **タスク名**`（Notionタスク粒度で集約）
   - サブ項目は `  - 詳細`
   - セクション: 「完了」「着手・進行中」「体調メモ」
4. ユーザーに今日のゴール・優先タスクを確認
   - 「持ち越しで〜がありますが、今日の優先度はどうしますか？」
   - 「今日のゴールを1つに絞るなら何にしますか？」
5. デイリーノートを更新

**テンプレート**:

```markdown
タグ: #daily-notes

# 📅 昨日やったこと（朝会用）

### 完了
- **<タスク名>**
  - 詳細

### 着手・進行中
- **<タスク名>**
  - 詳細

### 体調メモ
-

---

# 今日やること

> 体調メモ:

## 🎯 今日のゴール
- 主:
- α候補（時間と体調で判断）:

## 🔥 今日の優先タスク（優先度順）
- [ ] 1.

## 🔴 過去の未完了タスク
- [ ]

# 作業ログ


# 学んだこと / メモ


# 振り返り

```

### Mode R: 日次振り返り (daily-review)

**いつ**: 「振り返りしよう」「今日の振り返り」、1日の終わりに

**やること**:

1. **昨日の振り返りを確認**（前日のデイリーノートの振り返りセクション・ランクを読む）
2. 今日のデイリーノートを読む
3. git logで今日のコミット確認（**authorでフィルタ必須**）
   ```bash
   git log --all --oneline --since="today 00:00" --author="$(git config user.name)"
   ```
4. **ウィンドウ履歴を取得**（ActivityWatchが動いている場合）
   ```bash
   bash ~/.claude/skills/evaluation-coach/scripts/window-summary.sh
   ```
5. **稼働時間を評価期データに保存**（pmsetログは数日で消えるため、その日のうちに保存）
   ```bash
   bash ~/.claude/skills/evaluation-coach/scripts/save-today-work-hours.sh
   ```
6. サマリーを提示（完了/未完了/作業ログ/コミット/ウィンドウ履歴から抽出）
7. **昨日との比較**を提示（昨日の課題に対してどうだったか、ランクの変化）
8. **ユーザーに感想を聞く**（AskUserQuestion で選択肢を提示）
9. **成長ポイントをClaudeが提示**（観察した気づきを言語化）
10. **ランク評価を提示**（下記「ランク評価」参照）
11. **明日意識すること**を自然に提案
12. 最終的にまとめて「振り返り」セクションに記録:
    - 🤖 Claudeコメント（客観的な観察）
    - 📝 自分の振り返り（ユーザーの感想・気づき）
    - 🏆 今日の評価（ランク表）
    - 🌱 今日の成長ポイント
    - 🎯 明日意識すること

### Mode A: 収集 (collect)

**いつ**: 「今週どうだった」「最近の活動まとめて」など、まず事実を並べたいとき

**やること**:
1. `scripts/collect-week.sh [日数]` を実行（デフォルト7日）
2. git log（authorフィルタ）、merged PR、デイリーノート学び欄、retlaude reflections を統合表示
3. **ユーザーに見せる**: 「これが今週の活動です」

### Mode B: 接続 (connect)

**いつ**: 収集後、または「これってどの目標に効く？」と聞かれたとき

**やること**:
1. `.coach.yaml` の Objective/KR 一覧を読む
2. 収集した活動を1件ずつ「どのKRに効くか」分類
3. 弱い実績は「KR接続が弱い、何を足すと強くなるか」を明示
4. **ユーザーに確認**: 「この分類で合ってる？」

### Mode C: evidence化 (record)

**いつ**: KR接続が確定したら、もしくは「これは残しておきたい」とユーザーが言ったとき

**やること**:
1. `evidence/<Objective>/<YYYY-MM-DD>_<短い名前>.md` として書き出し
2. テンプレ（`evidence/_template.md`）に従い、何をしたか/判断/成果物/次アクション を埋める
3. 不足情報があれば**evidenceに「TODO」として残す**（嘘で埋めない）

### Mode D: コーチング (coach)

**いつ**: 「今期どう？」「足りないものある？」、または週次・月次の定期レビュー時

**やること**:
1. `.coach.yaml` の期間設定から現在の進捗位置（前半/中盤/後半）を判定
2. evidence ディレクトリ配下を走査し、KRごとのevidence件数・最終更新日を集計
3. 「KRごとのカバレッジ」「時期に対して足りない型のアクション」を指摘
4. 次の1〜2週間で取るべき具体行動を3つまで提案

### Mode E: 評価草案更新 (draft)

**いつ**: 月末・四半期末・「評価書き始めたい」と言われたとき

**やること**:
1. evidence全件を横断レビュー
2. `評価.md` のKR別セクションを更新（実績・判断ハイライト・波及効果）
3. 「これでは弱いKR」を明示し、残期間で補強すべきポイントを提示

### Mode W: 週次振り返り (weekly-review)

**いつ**: 「今週の振り返り」「週次レビュー」、金曜 or 月曜の自動起動時

**標準フロー**:

1. **collect**: 直近7日の活動を `collect-week.sh` で集約
2. **ウィンドウ履歴**: ActivityWatch で各日の作業時間配分を把握
3. **connect**: 各活動を KR にマッピング（曖昧なものはユーザー確認）
4. **record**: 残す価値のあるものを evidence/ に書き出し
5. **TIL分析**: 今週のTIL (`find ~/obsidian-vault/TIL -name "*.md" -mtime -7`) からパターン分析・改善アクション提案
6. **稼働時間集計**:
   ```bash
   bash ~/.claude/skills/evaluation-coach/scripts/work-hours-month.sh $(date +%Y-%m)
   ```
7. **リーダーシップ観点**でも振り返る（チーム貢献、委譲、作業者/リーダーのバランス）
8. **ランク評価**（週次はコメント付きで詳細に）
9. **coach**: 今週足りなかった観点・来週のフォーカスを提案
10. `.last_weekly_review` に日付書き込み（同週の再リマインダ抑止）
11. 記録先: `~/obsidian-vault/デイリーノート/YYYY/weekly/YYYY-Www.md`
12. デイリーノートの振り返り欄に「今週のevidence追加: N件 / 次週フォーカス: ...」を追記

## 月次レビュー

週次に加えて:
1. 残業時間集計 → `評価.md` の月次表を更新
2. KRごとの累積進捗を確認、目標からのズレを点検
3. 目標変更が必要そうなら「現行KRの問題 / 変更案 / 評価しやすさの改善 / 副作用」をまとめて提示

## ランク評価（S/A/B/C/D）

### 評価観点

| 観点 | 説明 |
|------|------|
| **成果量** | タスク完了数、PR数、コミット数など定量的なアウトプット |
| **品質** | コードの品質、レビュー状況、設計の明確さ、本人の納得感 |
| **リーダーシップ** | 作業者の枠を超えた行動。チーム貢献、基盤構築、委譲、レビュー支援、仕組み化 |
| **持続可能性** | 稼働時間、体力、ワークライフバランス |
| **問題解決力** | 課題の言語化→対策→実行のサイクル |
| **アピール価値** | 目標面談でアピールできる成果・行動ができているか |
| **総合** | 上記を総合した評価 |

### ランク基準

| ランク | 基準 |
|--------|------|
| **S** | 期待を大きく超えている |
| **A** | 期待を超えている |
| **B** | 期待通り |
| **C** | やや不足 |
| **D** | 大きく不足・要改善 |

### 評価のポイント

- **リーダーシップ**: 「作業者として手を動かす」だけでなく、チーム全体に影響する行動をしているか
- **アピール価値**: 目標面談で「こんなことやりました」と言える成果があるか（定量・定性両面）

### 記録フォーマット（日次）

```markdown
🏆 今日の評価
| 観点 | ランク |
|------|--------|
| 成果量 | B |
| 品質 | B |
| 持続可能性 | A |
| 総合 | B |
```

週次ではより詳細に、コメント付きで記録する。

## 成長支援のスタンス

**原則**: 質問攻めにしない。Claudeが観察して気づきを提示し、軽く確認する。

### やること

1. **観察して気づきを言語化する**
   - 「今日は〜ができていましたね」
   - 「昨日と比べて〜が改善していますね」
   - 「〜の部分、作業者を超えてリーダー的でしたね」

2. **AskUserQuestionで軽く確認**（質問攻めにしない）
   - 「今日の成長ポイント、私は〜だと思いましたが合ってますか？」
   - 選択肢を提示して選んでもらう形式も有効

3. **明日への意識付けを自然に**
   - 押し付けず、提案として

### やらないこと

- 質問攻め（「なぜ？」「どうする？」の連発）
- 熱すぎる励まし
- 成長を強制するような言い方

## やってはいけないこと

- **evidenceにない事実を補わない**: 評価文の見栄えのために盛らない。証拠が無いことは「無い」と書く
- **PRコメント投稿日と実績日を混同しない**: 活動日はevidenceのfront-matterに残す
- **テンプレート絶対主義にならない**: テンプレは出力安定のためのもの。実態に合わなければ更新する
- **OKRに過剰適合しない**: 評価のために仕事を歪めない。重要なのは「価値ある仕事を評価可能な形に変換すること」
- **ファイル編集は必ずEditツール**: bashの `>>` リダイレクトは使わない（文字化け防止）
